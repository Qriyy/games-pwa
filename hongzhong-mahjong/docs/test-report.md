#### BUG-011：canHu不支持副露胡牌 🔴 严重

**现象**：当玩家有副露（碰/杠）时，`canHu()` 无法正确检测胡牌。

**根因**：`HuDetection.canHu()` 内部调用 `tryHu(counts, wildCount, 5)` 时，`groupsNeeded` 始终传 5（= 4面子 + 1将），未减去已暴露的副露数量。

**影响**：有副露的玩家永远无法通过 `canHu` 检测胡牌。例如：
- 有2个碰副露时，手牌8张应检测 2面子+1将（groupsNeeded=3）
- 但实际传了 groupsNeeded=5，8张牌无法组成5个分组

**修复建议**：
```js
// hu-detection.js canHu() 中修改：
const meldsCount = melds ? melds.length : 0;
const normal = tryHu(counts, wildCount, 5 - meldsCount);
```

#### BUG-010：明杠未记录副露 🔴 严重

**现象**：执行明杠后，玩家手牌减少3张，但 `melds` 数组未增加杠的记录。

**根因**：`Actions.performGang()` 的明杠路径 (`tile >= 0`) 只做了：
1. 从弃牌堆移除杠牌
2. 从手牌移除3张同牌
3. 计算杠分

但**未执行** `st.melds[playerIdx].push(...)` 将杠记录到副露中。

而暗杠/补杠路径 (`tile < 0`) 正确地记录了副露。

**影响**：
- 明杠后，UI 渲染器无法显示杠的副露牌
- 碰碰胡检测 (`checkPengpenghu`) 无法识别明杠为合法面子
- 清一色检测可能遗漏明杠的花色信息

**修复建议**：在明杠路径末尾添加：
```js
st.melds[playerIdx].push({
  type: 'gang',
  tiles: [tile, tile, tile, tile],
});
```

#### BUG-012：AI canWin不检查红中 🟡 中等

**现象**：`AIModule.canWin()` 不要求手牌中包含红中即可判定为可胡牌。

**影响**：AI 模块的胡牌判断比游戏规则宽松。但由于实际游戏流程中，`canHu()`（游戏模块）先做检测，通过后才通知 AI，所以此 Bug 不会导致 AI 误胡。但 AI 的向听数/进张计算可能因此过于乐观。

**修复建议**：在 `AIModule.canWin()` 开头添加红中检查：
```js
function canWin(hand, exposedMeldCount) {
  if (countHongZhong(hand) === 0) return false; // 必须有红中
  // ... 其余逻辑
}
```

## 修复优先级建议

### 第一优先级（严重影响游戏逻辑）
1. **BUG-011**：`HuDetection.canHu()` — `tryHu` 的 `groupsNeeded` 应减去副露数
2. **BUG-010**：`Actions.performGang()` — 明杠路径应记录副露

### 第二优先级（AI行为不一致）
3. **BUG-012**：`AIModule.canWin()` — 添加红中检查以匹配游戏规则

### 第三优先级（防御性编程）
4. **BUG-013/014**：`Tiles.tileSuit()` — 添加越界校验
