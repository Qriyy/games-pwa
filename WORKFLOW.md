## 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 渲染 | HTML5 Canvas / DOM | 棋类用Canvas，牌类用DOM |
| 样式 | 内联CSS | 单文件方案，零依赖 |
| 逻辑 | 原生 JavaScript | 无框架，纯ES6+ |
| 音效 | Web Audio API | 无外部音频文件 |
| AI | Minimax / 启发式 | 棋类用Minimax，牌类用启发式 |
| 部署 | 单HTML文件 | 双击即玩，零配置 |

## 设计规范

### 视觉风格
- 暗色主题（#0a0a1a 背景）
- 金色/琥珀色点缀（标题、高亮）
- 中文字体优先（Microsoft YaHei）
- 棋盘绿色基调

### 交互规范
- 点击选中 → 再点确认（两步操作防误触）
- 快捷键：Ctrl+N 新局 / Ctrl+Z 悔棋 / Escape 关闭
- 移动端：触屏支持，passive:false 防滚动
- 动画时长：150-400ms

### 代码规范
- 单文件 HTML，全部内联
- 函数命名：camelCase
- 游戏状态用全局对象 G 管理
- 渲染函数统一叫 render()
- 音效函数统一在顶部定义

## 版本号规则

```
v主版本.次版本
v1.0  = 基础可玩版本
v1.1  = 功能增强（如加吃碰杠）
v2.0  = 大改版（如多人联网）
```

## 发版流程

1. 开发完成 → 自测通过
2. 更新 KANBAN.md 任务状态
3. 通知产品总监验收
4. 验收通过 → 标记完成

## 分工方式

当需要开发新游戏时：
1. **策划组**先行：输出规则文档
2. **前端组**主力：实现核心逻辑
3. **美术组**并行：UI和视觉设计
4. **测试组**收尾：功能和体验测试

多个子任务可通过 Hermes 的 delegate_task 并行派发给多个 Claude Code 实例。
